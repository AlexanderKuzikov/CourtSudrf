# CONTEXT.md — CourtSudrf

## Архитектурные решения

### 1. Почему не `go_search`

Эндпоинт `go_search` на sudrf.ru защищён JavaScript-челленджем (Qrator). Сервер проверяет
наличие HMAC-подписанной куки, которая устанавливается только после выполнения JS в браузере.

Используются эндпоинты без JS-защиты:

- **`ya_coords?type_suds=fs`** — возвращает все федеральные суды РФ одним JS-скриптом (2334 записи).
  Содержит код, название, адрес (не всегда), координаты. Параметр `type_suds` игнорируется —
  всегда возвращаются федеральные суды.

- **`ya_info&vnkod=X`** — возвращает детальную HTML-карточку суда по коду. Содержит: название,
  код, адрес (основной + ПСП через ";"), телефон, email, сайт.

- **`go_ms_search&ms_type=ms`** — возвращает всех мировых судей (7747) одной HTML-страницей.
  Содержит только код, название, регион и сайт. Адресов/телефонов/email в ответе нет.

### 2. Кодировка

Все ответы sudrf.ru в windows-1251. Декодируем через `iconv-lite`.

### 3. Разделение адресов

Поле адреса может содержать несколько адресов через ";".
Первый адрес — основной, остальные — ПСП. ПСП-адреса часто содержат пометку "(ПСП ...)".
Телефон целиком вешается на первый адрес (известное ограничение, см. BUG_REPORT B5).

### 4. Rate limiting

Bottleneck с 1.5 с между запросами. 2334 федеральных суда = ~58 минут.
Мировые судьи (MS) — один запрос, rate limiting не нужен.

### 5. Формат данных

Совместим с CourtHarvest2:
- `data/courts.json` — `{ meta, courts }` (как в CH2)
- `data/prefixes/{RRTT}.json` — чистый массив

Поля-расширения CourtSudrf (отсутствуют в CH2): `phone`, `email`, `addresses[]`, `court_type_name`.
Поля совместимости (всегда null в CourtSudrf): `okato`, `okmo`, `okpo`.

### 6. Приоритет полей

- **name**: приоритет у `ya_info` (свежие данные), а не у `ya_coords` — **но в коде наоборот**
  (см. BUG_REPORT B6).
- **address**: из `ya_info`. Если `ya_info` не ответил — из `ya_coords` (поле `adress` с опечаткой).
- **phone/email**: только из `ya_info`. Для MS — отсутствуют.

### 7. Мировые судьи (MS)

- Источник: `go_ms_search?ms_type=ms` — одна HTML-страница
- Данные: только код, название, регион, сайт
- Адреса/телефоны: отсутствуют на sudrf.ru. Брать из DaData (CourtHarvest2).
- Парсинг: регулярками по `<TR><TD>` блокам. Обработка ~1 с на 7747 записей.

### 8. Типизация кодов судов

Код = RRTTNNNN, где RR = регион (01-99), TT = тип (RS, OS, MS, GV...), NNNN = номер.

Маппинг кодов типов в `CourtRecord.COURT_TYPE_MAP` покрывает 15 типов (все известные).

### 11. ПСП (постоянные судебные присутствия)

ПСП — структурные подразделения судов в отдалённых населённых пунктах.

**Формат хранения:** суффиксные поля на корне объекта суда.
Плоский JSON, без вложенных массивов.

```json
{
  "code": "43RS0018",
  "name": "Котельничский районный суд Кировской области",
  "address": "612600, Кировская обл., г. Котельнич, ул. Советская, д. 27",
  "phone": "(83342) 4-07-57",
  "okmo": "33614000",

  "psp_count": 2,
  "psp_address_0": "612140, пгт. Даровской, ул. М. Гвардия, д. 21 А",
  "psp_phone_0": "(83336) 2-11-06",
  "psp_okmo_0": "33614404",

  "psp_address_1": "612270, г. Орлов, ул. Кирова, д. 65",
  "psp_phone_1": "(83365) 2-86-27",
  "psp_okmo_1": "33615404"
}
```

**Правила:**
- `psp_count = 0` — ПСП нет, суффиксные поля отсутствуют
- `psp_count = N` — есть N ПСП, заполнены поля с суффиксами `_0` … `_N-1`
- Корневые `address`/`phone`/`okmo` — основное помещение (main), совместимость с CH2
- Индексация строго от 0 до N-1, без пропусков
- При добавлении/удалении ПСП — переиндексация всех записей

**Какие поля дублируются на каждое ПСП:**
`psp_address_N`, `psp_phone_N`, `psp_okmo_N`.

**Ограничения и слабые места:**
1. Числовой лимит — заложен буфер 0..9 (макс 10 ПСП). При превышении — менять формат.
2. Размножение полей — каждое новое свойство ПСП добавляет N полей к объекту.
3. Хрупкость индексов — удаление ПСП требует переиндексации.
4. Валидация — `psp_count` должен строго соответствовать количеству заполненных суффиксных полей, иначе данные рассинхронизируются.
5. TypeScript-типизация — нужен генерируемый union-тип под макс. количество ПСП.

## Известные баги и ограничения

1. **B1: `all` command сломана** — не работает из-за неправильного вызова подкоманд.
2. **B2: `all` parseInt без лямбды** — приводит к NaN для `--delay` и `--limit`.
3. **B3: Bottleneck с minTime=0** — Warning при запуске `ms` команды.
4. **B5: Телефон не привязан к ПСП** — весь телефон на первом адресе.
5. **B6: Приоритет названий** — `ya_coords` затирает название из `ya_info`.
6. **Нет ИНН** — sudrf.ru не показывает ИНН (в отличие от DaData).
7. **MS без адресов** — на sudrf.ru нет адресов мировых судей.
8. **78 RS в CH2, но не в sudrf** — возможны расформированные/новые суды.
9. **Арбитражные суды** — отсутствуют на sudrf.ru (нужен отдельный источник/парсер).
10. **Commander + parseInt** — в Commander v13 нельзя использовать `parseInt` напрямую
    с default-значением. Нужна лямбда `(v) => parseInt(v, 10)`.
